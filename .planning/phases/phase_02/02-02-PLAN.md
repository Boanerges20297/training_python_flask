# Plan: 02-02 — Security Logic & Rotation

## Goal
Implement persistent blocklist logic and Refresh Token Rotation in the AuthService.

## Context
Decouple auth logic from the mock list and implement the "one-time use" logic for refresh tokens to prevent session hijacking.

## Wave 1: Persistent Blocklist Integration
- [ ] **Task 02-02-01**: Refactor `AuthService` to use `TokenBlocklist` model.
    - **Action**: 
        - Update `revoke_token(jti)` to create a `TokenBlocklist` entry.
        - Update `is_token_revoked(jti)` to query the database.
    - **Read First**: `backend/app/services/auth_service.py`
    - **Acceptance Criteria**: `MOCK_BLOCKLIST` is removed; DB is used instead.

- [ ] **Task 02-02-02**: Update `jwt_callbacks.py`.
    - **Action**: Ensure `check_if_token_is_revoked` calls `AuthService.is_token_revoked`.
    - **Read First**: `backend/app/jwt_callbacks.py`

## Wave 2: Refresh Token Rotation
- [ ] **Task 02-02-03**: Implement Token Rotation in `renew_access_token`.
    - **Action**: 
        - When a new access token is requested via refresh token:
            1. Revoke the *current* refresh token (add to blocklist).
            2. Generate a *new* refresh token.
            3. Return both new access and new refresh tokens.
    - **Acceptance Criteria**: Refreshing a token results in a new pair, and the old refresh token becomes invalid.
